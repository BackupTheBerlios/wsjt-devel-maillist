> WSPR:  I've backported the endian changes to the wspr branch (except for 
> getfile).  It's a pain to have diverged code like that; it would seem to 
> make sense to build wspr out of the trunk along side wsjt so that 
> algorithm improvements go into both.  

Yes, there are some pains related to the structure of our 
code tree.  Do keep in mind that at present we have some 
significant differences between what's wanted for WSPR (the 
program) and for WSPR-mode in WSJT.  Among these are the 
following:

1. WSPR uses 12000 Hz sampling.  WSJT, alas, uses 11025 Hz. 
  It would be better use 12000 Hz everywhere now, ... but 
that means a lot of changes in WSJT.  I haven't yet felt the 
time was right for making this major change.

2. For its beacon-like purposes, WSPR wants to decode all 
the signals it finds in a narrow band segment, currently 200 
Hz.  For making QSOs, WSJT may want to focus on signals 
within "Tol" of the "Freeze DF".
