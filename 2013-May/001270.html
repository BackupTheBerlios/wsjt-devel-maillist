Hi Edson and all,

I spent more time today thinking about how to keep WSJT-X from hanging 
if, for example, you try to start CAT control with the radio turned off. 
  Or if you're running with CAT control enabled, and then turn the radio 
off.

I have found no way to determine from hamlib calls that the 
communication path has been broken... This means that a rigOpen() or 
getFreq() may be issued, and we don't get an error code in return.  The 
function never returns, at all.  This seems to be true even if things 
like "timeout=1000" and "retry=3" are issues via setConf().

Can this really be true?  Does anyone know of a way around the problem? 
  Or do we just advise users to exit from WSJT-X before turning off the 
radio?
