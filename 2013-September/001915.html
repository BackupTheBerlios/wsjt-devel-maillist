On 05/09/2013 18:22, Joe Taylor wrote:
Hi Joe & John,
> Hi John,
>
> Yes, we should definitely go to 48 kHz sample rate for audio input.
>
> In your test code, why did you switch from using fil1.f90 (a routine 
> from the WSPR repository) to using Bill's code that simply adds four 
> samples?  The simple method is faster, to be sure, but I'd be 
> surprised if fil1 is so slow that it matters much.
It would be even better if it had a small enhancement to work on 
interleaved multichannel data. I think we can probably say that we only 
ever want one channel down sampled so a version that could select n'th 
of m channels would drop in cleanly.
>
> The problem with the sum-of-4 approach is that any signals or noise in 
> the range 6-24 kHz will be aliased into the downsampled data.  The 
> fil1() approach is better since it applies an excellent low-pass 
> filter before the decimation-by-4 downsampling.  Of course, most of 
> the radios we're using shouldn't pass much of anything in that range 
> to the audio output, so you're usually OK. Still, we might as well do 
> it right.
Joe, my non-DSP brain keeps telling me that averaging every 4 samples is 
somehow a low pass filter. Can you explain in words that I might 
understand why it still folds down signals from the upper half of the 
usable bandwidth?
>
> Parameters in the present fil1() are not exactly what we need for 
> WSJT-X.  The LPF cutoff frequency is 3 kHz; we'd want to move it up to 
> 4 kHz or a bit higher.  The stopband attenuation is 60 dB; we could 
> relax this to, say, 40 dB.  The result will be an FIR filter with a 
> few more taps than the 37 used in fil1(), but I think speed would 
> still be OK.
>
> If you have code that already implements the fil1() approach, I can 
> easily produce the code for a slightly modified version of fil1.f90.
>
>     -- Joe, K1KT
