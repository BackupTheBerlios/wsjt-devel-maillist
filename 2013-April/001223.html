Hi Joe,

I fully agree with your enumeration of tasks. One suggestion I have would
be to elect a production release after some testing time to allow us to
polish edges and eliminate serious bugs. We could have a list of features
to work on and once that is complete, we could switch to polishing and bug
hunting mode and once stability is achieved, we could promote a revision to
production. The production code base could then be used to build packages
and pre-compiled binaries. What do you think?

As you have stated, JT9-1 is becoming popular very fast on HF bands.
However, I am not 100% sure the other modes don't have their niche. Perhaps
the reason that they are not popular yet is that low band experimenters are
not yet familiar with them. I, myself, have introduced the JT9-X to a
couple of hams that like operating QRP on 160m. They were not at all aware
of the new modes. So... Since the current code base has all the modes
implemented, may be we could hold onto them for a little longer. We could
try to remediate the memory requirements issue with some programming.
Perhaps a command line argument for which mode to run -- having JT9-1 as a
default mode?

73,

-- Edson PY2SDR


On Tue, Apr 23, 2013 at 12:49 PM, Joe Taylor <joe at princeton.edu> wrote:

> I'd like to consult others on the general topic of "What needs to be done
> before a formal release of WSJT-X Version 1.0?"
>
> As a bare minimum, my own short list includes the following essentials:
>
> 1. Complete recent work on rig control via calls to hamlib functions.
>
> 2. Track down and fix a recently reported bug that causes erratic crashed
> of the GUI.
>
> 3. Write a full User's Guide to replace the present early draft.
>
> I'm presently working on all of these.  Are there other important program
> features that should be added, before a more formal release?
>
> Please feel free to submit your (preferably prioritized) list!
>
>
> One more thought.  As I'm sure you are aware, 99.9% of the use of WSJT-X
> is with the JT9-1 mode.  The other submodes (JT9-2 through JT9-30) were
> included as "specialty modes", possibly useful on the MF and LF bands.
> There is little reason to use them at HF.
>
> The WSJT-X code could be made somewhat simpler (and much smaller in memory
> requirements) if submodes JT9-2, ... JT9-30 were eliminated from the code
> going forward.  Do you have an opinion about the advisability of doing so?
