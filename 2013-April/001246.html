Hi Greg,

I am sure that we (I, for sure!) could benefit from your advice on 
packaging WSJT-X and its sister programs.

We have taken some care to be sure that the WSJT-X code compiles and 
runs properly in Windows and Linux; we don't have anyone currently 
compiling the program in OS X or FreeBSD.  I make the Windows packages 
myself, simply using Inno Setup to bundle the executable and necessary 
support files into a Windows "setup.exe" file.  We don't yet have a 
standard packaging scheme for *nix.

If you're interested, we'd be very happy to have your help and/or advice 
in devising a better system.

>  From a packager viewpoint (I am one of the maintainers of pksrc, a
> multi-OS multi-CPU packaging system), the important thing is to have a
> clean and straightforward build that follows normal procedures.  Then,
> creating a package for almost all packaging systems is pretty easy.  So
> that means:
>
>    in a README of some sort, describe the dependencies clearly, without
>    assuming much about the packaging system in use.  (The notion of
>    "development packages" to get include files and static libraries is a
>    linuxism; pkgsrc and other systems used on *BSD do not have this
>    split.)
>
>    be able to ./configure, make, make install, using DESTDIR (to put the
>    installed files in a staging area; many packaging systems do that to
>    avoid having to be root to build and to avoid modifying the host
>    system).   autoconf isn't necessary, but the closer a program is to
>    one of the established mechanisms, the better.  packaging systems have
>    support for interacting with standard approaches.
>
>    Install everything, including documentation, that the user is expected
>    to want to look at.  Include configure-time switches so that docs and
>    examples can be put in the directory that the packaging system's
>    placement rules expect.
>
>    Ideally cross-compilation would be supported, via the normal autoconf
>    method.
>
