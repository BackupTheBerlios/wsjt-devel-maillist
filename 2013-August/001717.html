On 06/08/2013 17:13, Joe Taylor wrote:
> Bill --
Hi Joe,
>
> I think your proposal could be workable.  What to do when we want to 
> terminate a transmission early, for some reason?  I guess you can 
> force a change to idle state, as needed?
Yes, it's just as simple as telling the modulator go to idle and signal 
that change. This is easy now it runs in the GUI thread.
>
> One concern: this seems like rather a lot of changes just to solve a 
> problem that I think (a) appears only in Linux, and (b) has other 
> possible solutions.
I agree in principle, but I see a lot of code in the MainWindow that 
wouldn't be necessary (or would be hidden away in the implementation of 
some cooperating classes) if we had some signals coming back from the 
transmit process to drive the GUI state.

I do have an ulterior motive. I'd like to see the GUI state handling 
move to a more event driven model. I find it very hard to understand the 
flow where there is a mix of a lot of sequential code intermingled with 
event handling code. Qt has a nice state machine implementation which 
integrates very well with signals and properties and seems absolutely 
suited to the JT modes which are strict state machines themselves. By 
implementing or re-factoring functionality slowly towards object classes 
that are driven by slot and signal events with properties to parametrize 
their behaviour; we wouldn't be pushing uphill so much against Qt.

I am testing some minor changes the new Modulator to test this 
sequencing approach without actually changing the tx logic in the 
mainwindow code.
