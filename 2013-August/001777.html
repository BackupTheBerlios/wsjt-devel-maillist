On 08/09/2013 01:41 AM, Bill Somerville wrote:

> On 09/08/2013 00:19, Josh Rovero wrote:

>> Both the receive and transmit streams in r3537 have names like
>> QtmPulseContext, and the transmit audio level needs to be reset at
>> each transmission.  That keeps me a lot busier than I'd like.......
> Yes this is a fairly serious issue. At the moment I have coded the sound
> output to start a new stream every period. I didn't want to do it this
> way but at the moment I don't have a way of reliably resuming a
> suspended stream when needed. The best alternative I have so far is to
> continuously stream audio output of silence between tx periods.
>
> This will still leave the issue that every time the application is
> restarted a new stream will be created with a new name. See the thread
> below for some more details including why we can't set the stream name.
>
> https://lists.berlios.de/pipermail/wsjt-devel/2013-August/001711.html
>
> I intend to have another try at keeping a single stream for output
> throughout a application session.
>
> The other option is to fit an output volume control to wsjt-x, that we
> could persist across sessions too.

I think that it is important to always have the same name of the RX and 
TX streams, so that the volume does not have to be set again at every 
start of the program. When appropriate plugins are used with pulseaudio, 
the volume of each streams will be remembered.
