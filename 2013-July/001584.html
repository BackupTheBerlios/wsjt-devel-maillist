>  Yesterday I've done a few tests with Qt5 Audio and have some
>> observations. On Linux, Qt5 either uses pulseaudio or ALSA as sound
>> input/output. Either one or the ohter (not both).
>>
>
> The idea behind pulseaudio is to liberate the application from the need to
> deal with the hardware. The application, as client, requests streams from
> the pulseaudio server. The selection of devices, the mixer, the volume
> control are now a matter which pulseaudio has to manage. There are tools
> available to the user. pulseaudio is now the standard sound system of most
> modern Linux distributions.
>

I thought that was the purpose for ALSA! ;-)

I understand the idea behind pulseaudio as a sound server. However, I had
some bad experience with it in the past while trying to use it with fldigi
and with SDR software. It added unnecessary latency, complexity and, at the
time, it did not work well with jackd. Also, there is the risk of some
application injecting undesirable audio samples on a stream that is being
transmitted. As far as I know, pulseaudio is (was) *not* designed for
real-time audio. This is fine for using the computer to play music, but
undesirable for low latency audio applications. Jackd shines in this
respect.

We could, of course, run some tests with recent pulseaudio releases and Qt5
and check if it is good enough for WSPR-X.
