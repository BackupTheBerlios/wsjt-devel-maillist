Thanks John.

Yes, I know that I need to save and restore the settings to/from wspr.ini.

There are two reasons I didn't do it yet (apart from doing the more
important things first).

The most important reason is that I wanted to initially perturb the existing
code as little as possible, so as not to introduce bugs.

The second is to let the feature stabilize first, let users test it, see if
we need to make changes to the user interface, and only then modify the
structure of the ini file. I wanted to avoid creating files on users'
computers that would then be incompatible with future versions of wspr (e.g.
different parameter names). It's probably mostly paranoia, since if wspr.py
would not recognize some old parameter name it would probably just ignore
it, but I didn't look at that part of the code yet so I wanted to be on the
safe side.

I still think that it makes sense to let users test this feature first
without saving to a file. Once everybody is happy, a follow-up release will
add save/restore.
